## Ideas & Reasoning

### Core Architecture Decision
- **Verdict is code, not LLM.** The deterministic eligibility engine (`check_refund_eligibility`) computes APPROVE/DENY/ESCALATE. The LLM handles language and orchestration only. This is un-jailbreakable — no prompt injection can override a Python `if` statement.
- **Defense in depth:** Hardened system prompt (layer 1) + tool-level guard on `issue_refund` re-validates eligibility (layer 2) + deterministic engine (layer 3).

### Technology Choices
- **LangGraph 1.2+** for agent loop — explicit state machine maps 1:1 to trace UI. Uses prebuilt `ToolNode` + `tools_condition`.
- **FastAPI** for API — async, typed, auto OpenAPI docs.
- **React 19 + Tailwind v4 + Vite 8** for frontend — zero-config CSS, fast HMR.
- **JSON-based data store** (not SQLite) — simpler, no ORM overhead for 15 customers. Loaded once into memory.

### Data Design
- 15 customers with deliberate edge cases baked in:
  - CUST-001/ORD-1002: final_sale item (R1 test)
  - CUST-002/ORD-2001: $529.98 order (R2 escalation test)
  - CUST-003/ORD-3001: delivered March 5, out-of-window (R3 test)
  - CUST-004/ORD-4001: already refunded (R5 test)
  - CUST-006/ORD-6001: damaged + final_sale leather jacket (R6 override test)
  - CUST-011/ORD-11001: $650 handbag (second R2 test)
  - CUST-013/ORD-13001: final_sale winter coat (second R1 test)

## Decisions Made

### 02:15:31 — Build Strategy
Starting Ralph Loop for Phase 0 scaffold. Project has only docs — no backend/frontend code exists. Following fuzzy-zooming-platypus.md build plan (8 phases).

### Phase Ordering
Built in strict dependency order: Data → Engine → Tools → Agent → API → UI → Tests. This means the hardest correctness work (eligibility engine) is done and tested deterministically before any LLM is added.

### Rule Evaluation Order
R4 (ownership) → R5 (already refunded) → R1 (final sale, with R6 damaged exception) → R3 (refund window) → R2 (>$500 escalation) → APPROVED. Hard denials checked before escalation so strictest rule wins.

### JSON vs SQLite
Chose in-memory JSON store over SQLite. 15 customers is tiny — SQLModel/SQLite adds complexity without benefit. Data loads once via `_load_data()` lazy init.

### Error Handling Improvement
After discovering OpenAI 429 quota errors returned as generic 500, added specific HTTP error handling in chat endpoint:
- 503 for missing API key
- 503 for quota exceeded
- 503 for invalid key
- 500 for other agent errors

### npm SSL Issue
DaVita corporate proxy causes `SELF_SIGNED_CERT_IN_CHAIN` for npm. Fixed with `npm config set strict-ssl false`.

### Phase 3 — Tools
- `backend/app/tools/tools.py` — 7 `@tool`-decorated functions:
  1. `lookup_customer(identifier)` — find by email/ID/name
  2. `list_orders(customer_id)` — all orders for customer
  3. `get_order(order_id)` — single order
  4. `get_refund_policy()` — returns policy markdown
  5. `check_eligibility(order_id, customer_id)` — deterministic verdict
  6. `issue_refund(order_id, customer_id)` — **re-validates eligibility internally**, rejects if not APPROVED
  7. `escalate_to_human(order_id, reason)` — mock escalation with ticket ID
- Exported `AGENT_TOOLS` list for LangGraph

### Phase 4 — Agent Loop + Prompt
- `backend/app/agent/prompts.py` — Hardened system prompt with 5 non-negotiable rules
- `backend/app/agent/llm.py` — Swappable LLM client (OpenAI/Anthropic via env var)
- `backend/app/agent/graph.py` — LangGraph StateGraph:
  - `AgentState` extends `MessagesState` with trace, decision, run_id
  - Nodes: agent → tools (ToolNode) → extract_decision → agent
  - `tools_condition` for routing (tools or END)
  - `extract_decision` captures APPROVED/DENIED/ESCALATED from tool outputs

### Phase 5 — Tracing + API
- `backend/app/trace/tracer.py` — `TRACE_STORE` (defaultdict), `set_current_run()`, `trace_step()` context manager capturing latency_ms, tokens_in, tokens_out, retries, error
- `backend/app/main.py` — FastAPI app:
  - `GET /api/health` — liveness check
  - `POST /api/chat` — invokes agent graph, returns ChatResponse with reply + decision + trace + run_id
  - `GET /api/runs` — list all runs with step count, total latency, total tokens
  - `GET /api/runs/{run_id}/trace` — full trace for a run
  - CORS locked to `http://localhost:5173`
  - Error handling for missing API key (503), quota exceeded (503), invalid key (503)
- `backend/app/config.py` — env-driven settings
